# JLance V2.1 Nested Type Implementation — Audit Report vs. Canonical Rust Lance

**Date:** 2026-04-19  
**Scope:** `lance-format/src/main/java/com/github/jlance/format/` (reader only)  
**Reference:** `lancedb/lance` `rust/lance-encoding/src/repdef.rs` and related structural decoders  
**Tests passing:** 244 / 244 (M01–M58)

---

## Executive Summary

The Java port covers the **happy path** for `MiniBlockLayout` and `ConstantLayout` nested types well enough to pass M50–M58.  However, a line-by-line comparison with the canonical Rust implementation reveals **six categories of divergence** ranging from cosmetic debug output to missing core algorithms.  The most serious gaps are:

1. **No `unravel_validity` equivalent** — nullable struct/item validity inside multi-layer lists is reconstructed ad-hoc rather than by the Rust `RepDefUnraveler` algorithm.
2. **No `CompositeRepDefUnraveler`** — multi-page/multi-block rep-def merging is a hand-rolled short-array concatenation that does **not** re-run the unraveler across block boundaries.
3. **`FullZipLayout` lists are entirely unsupported**.
4. **Debug artifacts** (disk writes, `System.out.println`) remain in production code.
5. **`ConstantLayout` `NULL_AND_EMPTY_LIST` is hard-coded to "all null"**.
6. **Missing `levels_to_rep` lookup table** — Java uses a simplified `maxLevel` check that happens to work for current tests but diverges from the Rust design.

None of the gaps above trigger failures on the existing 244 tests, but they are **latent bugs** that will surface with richer test data (e.g. multi-page `FullZipLayout`, `NULL_AND_EMPTY_LIST` with non-null data, deeply interleaved struct/list pages).

## 1. Rep/Def Unraveler (`RepDefUnraveler` vs. `V21ListUnraveler`)

### 1.1 Architecture Divergence

| Aspect | Rust (`RepDefUnraveler`) | Java (`V21ListUnraveler`) | Risk |
|--------|--------------------------|---------------------------|------|
| **Core data** | `rep_levels: Option<LevelBuffer>`, `def_levels: Option<LevelBuffer>` | `short[] repLevels`, `short[] defLevels` | Low — direct translation |
| **Layer semantics** | `def_meaning: Arc<[DefinitionInterpretation]>` | `List<RepDefLayer>` | Low — protobuf enum maps 1:1 |
| **`levels_to_rep` table** | ✅ Built in `new()`; maps every def level to the rep level at which it is visible | ❌ Absent; Java uses a flat `maxLevel` integer | **Medium** — `maxLevel` is equivalent for current tests, but the Rust table handles edge cases (e.g. `NullableItem` after a list layer) more precisely |
| **State tracking** | `current_def_cmp: u16`, `current_rep_cmp: u16`, `current_layer: usize` | `currentDefCmp: int`, `currentRepCmp: int`, `currentLayer: int` | Low — identical |
| **`num_items`** | Stored for the "no def levels" fast path | Not stored | Low — Java recalculates on demand |

### 1.2 `unravel_offsets`

**Behavioral parity:** ✅ High for the code paths exercised by M50–M58.

**Notable differences:**

1. **Invisible entry handling**
   - Rust: `let visible_at = self.levels_to_rep[def as usize]; if visible_at > self.current_rep_cmp { /* invisible, keep in buffer */ }`
   - Java: `if (defVal > maxLevel) { /* invisible, keep in buffer */ }`
   - *Analysis:* For the layer stacks generated by the current Python test generators the two checks are equivalent.  If a future test introduces `NullableItem` layers **after** a list layer inside the same column, the Rust `levels_to_rep` will still be correct while the Java `maxLevel` loop (which stops at the first subsequent list layer) may miscount.

2. **`def == 0` fast path**
   - Rust: explicit `if def == 0 { /* valid list */ }` as the **first** check after the invisible guard.
   - Java: same, added during the M50–M58 fix.  Before that fix the Java code checked `defVal == nullLevel` **before** `defVal == 0`, which caused `EmptyableList` / `AllValidList` (where `nullLevel == 0`) to misclassify valid entries as null.
   - *Status:* Fixed and verified.

3. **Buffer truncation**
   - Rust truncates by adjusting a slice length (zero-copy).
   - Java allocates new `short[]` arrays and `System.arraycopy`.
   - *Risk:* Low — correctness is the same; performance differs but is not the focus of this audit.

### 1.3 Missing: `unravel_validity`

Rust `RepDefUnraveler` has a second major method:

```rust
pub fn unravel_validity(&mut self) -> Result<Option<BooleanBuffer>>
```

This method is called **after** all list layers have been consumed.  It reconstructs the validity bitmap for the **innermost primitive or struct** by scanning the remaining (truncated) def levels.  The algorithm:

1. Skips remaining non-list layers (just like `unravel_offsets`).
2. If the current layer is `NullableItem`, builds a `BooleanBuffer` where `def == current_def_cmp` means valid and `def == current_def_cmp + 1` means null.
3. If there are no nullable layers, returns `None` (all valid).

**Java equivalent:** ❌ **Missing entirely.**

Java handles primitive validity in three different ad-hoc places:
- `MiniBlockLayoutDecoder.decode()` builds chunk-level validity bitmaps during data decoding.
- `decodeV21ListColumn()` (single-layer path, lines 1069–1118) manually walks rep/def arrays to set `itemValidity[]` and then calls `itemVec.setNull(i)`.
- `buildStructVector()` (lines 470–503) derives struct validity from the **first child's** def levels using `computeNullItemLevel()`.

*Impact:* The Java approach works for the tested layouts because `MiniBlockLayoutDecoder` happens to decode validity alongside data.  If a layout stores **only** rep/def levels with no inline validity bitmap (e.g. a future `FullZipLayout` with `NullableItem`), the Java code has no standalone validity-reconstruction path.

### 1.4 Missing: `max_lists()`

Rust provides:

```rust
pub fn max_lists(&self) -> usize
```

This returns the worst-case number of lists at the current layer (equal to `rep_levels.len()` when every rep level starts a new list).  It is used by upstream decoders to pre-allocate Arrow buffers.

**Java:** ❌ Not present.  Java uses the caller-supplied `numRows` for allocation, which is correct only when the caller already knows the row count from the page metadata or parent structure.

## 2. Multi-Page / Multi-Block Rep-Def Merging

### 2.1 Rust: `CompositeRepDefUnraveler`

The Rust codebase has a `CompositeRepDefUnraveler` (mentioned in the `repdef.rs` module docs) that combines **multiple sets of rep-def buffers** (e.g. from multiple mini-blocks or multiple pages) and treats them as a single logical stream.  The composite unraveler:

1. Concatenates the level buffers.
2. Re-runs `unravel_offsets` / `unravel_validity` across the **merged** stream.
3. Produces a single offsets/validity result.

This is critical because each mini-block or page may have its own local rep/def compression, and simply concatenating Arrow vectors page-by-page is **not** sufficient — the offsets at page boundaries must be adjusted.

### 2.2 Java: `mergeShortArrays` + `VectorBatchAppender`

Java handles multi-page columns in `decodePrimitiveColumn()` (lines 510–551) and `decodeV21ListColumn()` (lines 948–950):

```java
short[] repLevels = mergeShortArrays(allRepLevels);
short[] defLevels = mergeShortArrays(allDefLevels);
```

`mergeShortArrays` is a simple `System.arraycopy` concatenation.  For the **data vectors**, Java uses `VectorBatchAppender.batchAppend()`.

**Divergence:**
- ✅ **Data vectors:** `VectorBatchAppender` correctly handles offsets and validity across pages for flat arrays.
- ⚠️ **Rep/def levels:** Concatenating short arrays is correct **only if** the levels are independent per page.  For V2.1 `MiniBlockLayout` where each mini-block has its own compressed rep/def stream, the Java code decodes each block independently inside `MiniBlockLayoutDecoder.decode()` and then appends the decoded vectors.  The rep/def levels are not used for cross-page offset reconstruction in the current code path.
- ❌ **List offsets across pages:** If a list spans a page boundary, the Java `unravelOffsets` is called on the **merged** rep/def arrays.  This is **superficially** correct, but the Rust composite unraveler would also verify that the first entry of each subsequent page has the correct rep level to continue the previous list.  Java does not perform this validation.

*Impact:* Currently low because all test files have a single page per column.  Will become a bug when `max_page_bytes` forces multi-page V2.1 list columns.

## 3. Struct Decoding

### 3.1 Rust Approach

Rust uses `StructuralStructDecodeTask` (not fully explored in this audit, but referenced in `repdef.rs` docs).  The key design points:

1. A struct **shares** parent rep/def information with all its children.
2. Each child column is decoded **independently** but with the **same** starting rep/def state.
3. After decoding, the struct's own validity is derived from the **intersection** of child validities (or from a dedicated `NullableItem` layer if present).

### 3.2 Java Approach (`buildStructVector`)

Java `buildStructVector()` (lines 422–508):

1. Creates an empty `StructVector`.
2. For each child, calls `readField()` sequentially.
3. After each child, **re-clones** `parentRepLevels` / `parentDefLevels` if the child consumed them (lines 461–467):
   ```java
   if (parentRepLevels != null && childArray.repLevels != null
       && childArray.repLevels != parentRepLevels) {
     parentRepLevels = parentRepLevels.clone();
     ...
   }
   ```
4. Derives struct validity from the **first child's** def levels (lines 470–503).

**Divergence:**
- ✅ Re-cloning ensures each child sees the original state.  This matches Rust's shared-parent design.
- ⚠️ Struct validity from **first child only** is a heuristic.  Rust would compute the intersection of all children.  If the first child is all-valid but a sibling has nulls, the Java struct may incorrectly mark the row as valid.
- ⚠️ The `computeNullItemLevel()` helper (line 1528) returns the def level that indicates "null struct item".  It scans layers inner-to-outer and returns at the first `NullableItem`.  This is correct for simple cases but does not handle stacked `NullableItem` layers (e.g. `nullable struct<nullable int>`).

*Impact:* No test currently exercises struct validity with mixed child nullability.  M55 tests nullable inner fields (`nullable list<int>`, `nullable string`) but the struct itself is not nullable in those tests.

## 4. List Decoding: `decodeV21ListColumn`

### 4.1 Layer-Count Logic

Java `decodeV21ListColumn()` decides whether to use the multi-layer path with this condition (line 978):

```java
if (listLayerCount > 1 || itemField.getType() instanceof ArrowType.Struct || itemIsList) {
```

This means:
- Multi-layer lists (e.g. `list<list<int>>`) → multi-layer path.
- Struct items (e.g. `list<struct<...>>`) → multi-layer path.
- Nested list items (e.g. `list<list<int>>` where inner list is the item) → multi-layer path with `skipLayers = 0`.

**Rust equivalent:** The Rust decoder does not have a single monolithic `decodeV21ListColumn`.  Instead, it uses a recursive `StructuralListDecodeTask` that calls `unravel_offsets` for **each** list layer as it descends/ascends the tree.  The Java approach of pre-computing all `UnravelResult`s and then building inside-out is functionally equivalent but less modular.

### 4.2 `skipLayers` and `computeUnravelerStartState`

Java has `skipListLayers()` (line 377) and `computeUnravelerStartState()` (line 1267) to handle the case where an inner list is decoded by a separate column (multi-column list fallback).

**Rust:** No direct equivalent — Rust handles this by creating a fresh `RepDefUnraveler` for each column with the correct starting state derived from the field hierarchy.

### 4.3 ConstantLayout Nested Lists

Java has a workaround for `ConstantLayout` nested lists (lines 1020–1030):

```java
if (itemIsList && hasConstantLayout && !layerResults.isEmpty()) {
    int missingLayers = countListLayersDeep(field) - layerResults.size();
    for (int m = 0; m < missingLayers; m++) {
        ... // inject empty inner layer results
    }
}
```

This is a **heuristic** that assumes missing inner layers are all-empty.  It works for M56/M57 (deep `list<int>` with `ConstantLayout`) because the inner layers are indeed empty.

**Rust:** The Rust `ConstantLayout` path does not need this hack because `RepDefUnraveler` processes layers one at a time, and if a layer has no rep/def data it simply produces empty results naturally.

## 5. Layout Support Gaps

### 5.1 `FullZipLayout`

| Feature | Rust | Java |
|---------|------|------|
| `NULLABLE_ITEM` primitive | ✅ | ✅ |
| Repetition levels | ✅ | ❌ Throws |
| Variable width | ✅ | ❌ Throws |
| Multi-level def | ✅ | ❌ Throws (`bitsDef != 1`) |
| List layers | ✅ | ❌ Not handled in `decodeV21ListColumn` |

*Impact:* `FullZipLayout` is used for large fixed-width pages (e.g. large `int` arrays).  If a user stores nested lists with `FullZipLayout`, the Java reader will crash with `UnsupportedOperationException`.

### 5.2 `BlobLayout`

| Feature | Rust | Java |
|---------|------|------|
| Large binary data | ✅ | ✅ |
| With rep/def layers | ✅ | ❌ Throws |

*Impact:* Blob columns with structural encoding (e.g. `list<binary>`) are unsupported.

### 5.3 `ConstantLayout` `NULL_AND_EMPTY_LIST`

Java hard-codes this as **all-null** (lines 336–346):

```java
case REPDEF_NULL_AND_EMPTY_LIST:
    // For constant layout with NULL_AND_EMPTY, we assume all null
    ...
    validity[i] = false;
```

**Rust:** `ConstantLayout` stores explicit `num_rep_values` and `num_def_values`.  If both are zero, the array is truly empty (no rows).  If they are non-zero, the actual rep/def levels are decoded and interpreted normally.  The "all null" assumption is only valid for the zero-values case.

*Impact:* M42 (ConstantLayout with `NULL_AND_EMPTY_LIST` and actual data) would fail if tested.

## 6. Code Quality & Debug Artifacts

### 6.1 Active Debug Prints

| Location | Frequency | Content |
|----------|-----------|---------|
| `LanceFileReader.java:1393` | Every `buildListVector()` call | `field name`, `numRows`, `childClass`, `childCount` |

*Fix:* Remove or replace with SLF4J `logger.debug(...)`.

### 6.2 Disk-Writing Debug Code

```java
// LanceFileReader.java:619-630 (inside decodeListColumn, V2.0 fallback)
java.nio.file.Files.write(
    java.nio.file.Path.of("C:/Users/22591/jlance/debug_buffer_" + i + ".bin"),
    buffers.get(i));
```

*Impact:* Writes to the filesystem on **every** V2.0 list decode.  In production this would be a security/performance issue.
*Fix:* Remove entirely.

### 6.3 Silent Exception Swallowing

```java
// MiniBlockLayoutDecoder.java:49-58
try {
    repLevels = extractRepetitionLevels(...);
} catch (Exception e) {
    // Ignore extraction failures for rep levels
}
```

*Impact:* If rep/def extraction fails, the decoder silently continues with `null` levels, which will likely produce incorrect data rather than a clean error.
*Fix:* Re-throw or at least log the exception.

### 6.4 `hasConstantLayout` Overwrite Bug

In `decodeV21ListColumn()`, `hasConstantLayout` is computed before the page loop (line 797) but then **overwritten** inside the loop (line 841):

```java
for (ColumnMetadata.Page page : cm.getPagesList()) {
    ...
    hasConstantLayout = layout.hasConstantLayout(); // overwrites outer check
```

If a column has **mixed** page layouts (e.g. first page `MiniBlockLayout`, second page `ConstantLayout`), the outer `hasConstantLayout` flag is meaningless and the inner overwrite may cause the post-loop logic to use the wrong path.

*Impact:* Currently theoretical — no test has mixed-layout pages.
*Fix:* Remove the outer `hasConstantLayout` flag or rename it to `allPagesConstantLayout`.

## 7. Boundary Cases Not Covered by Tests

| Scenario | Rust Behavior | Java Behavior | Likely Outcome |
|----------|---------------|---------------|----------------|
| `list<struct<a: list<int>, b: list<int>>>` — struct with **two list children** | Each child has its own column and rep/def stream; composite unraveler handles both | `buildMixedNestedVector` processes children sequentially starting from the same `layerIndex`. Both children would try to consume the **same** `layerResults` element because `layerIndex` is passed by value. | **Bug** — second child would get wrong offsets or crash with "Not enough layer results". |
| `nullable struct<nullable int>` — stacked `NullableItem` | `NullableItem` layers accumulate; `def == 2` means null struct | `computeNullItemLevel` returns at the **first** `NullableItem`, so `def == 1` is treated as null struct. | **Bug** — struct nullity would be misclassified. |
| `FullZipLayout` with `list<list<int>>` | Supported | `decodeV21ListColumn` throws at line 939 | **Crash** |
| Multi-page V2.1 list with page boundary inside a list | `CompositeRepDefUnraveler` adjusts offsets across blocks | Simple short-array concatenation; offsets may be wrong at boundary | **Potential bug** — untested |
| `ConstantLayout` `NULL_AND_EMPTY_LIST` with non-zero values | Decodes actual rep/def levels | Hard-coded to all-null | **Bug** — M42 would fail |
| `LargeList` with `FullZipLayout` | Supported | Throws in `FullZipLayoutDecoder` | **Crash** |
| Struct validity when first child is all-valid but second child has nulls | Intersection of all children | Only checks first child | **Bug** — struct may show valid when it should be null |

## 8. Recommendations (Priority Order)

### 🔴 P0 — Fix Before Next Release

1. **Remove disk-writing debug code** (`decodeListColumn` line ~619).
2. **Remove or demote `buildListVector` debug print** (line 1393) to SLF4J debug.
3. **Fix silent exception swallowing** in `MiniBlockLayoutDecoder` — at minimum log the error.

### 🟡 P1 — Fix When Adding M59+ or FullZip Tests

4. **Implement `unravel_validity` equivalent** in `V21ListUnraveler` or as a standalone helper.  Replace the ad-hoc validity reconstruction in `decodeV21ListColumn` single-layer path.
5. **Add `levels_to_rep` lookup table** to `V21ListUnraveler` to match Rust's visibility logic exactly.
6. **Fix `hasConstantLayout` overwrite** — use `boolean allPagesConstant = ...` computed once before the loop.
7. **Fix `ConstantLayout` `NULL_AND_EMPTY_LIST`** — do not assume all-null when `numRows > 0`.

### 🟢 P2 — Architectural Improvements

8. **Add `CompositeRepDefUnraveler` equivalent** for robust multi-page/multi-block list decoding.
9. **Support `FullZipLayout` repetition levels** — this is a significant piece of work (~200 lines in Rust).
10. **Improve struct validity derivation** — compute intersection of all children, not just first child.
11. **Add M59 Java test** (`list<struct<list<struct<list<list<int>>>>>>`) to exercise 6-layer mixed nesting.
12. **Add test for struct with two list children** (see boundary case #1 above).
