# JLance Agent Guide

This document describes the JLance project for AI coding agents. Read this first before making any changes.

## Project Overview

**JLance** is a Java re-implementation of the [Lance data format](https://lancedb.github.io/lance/) reader. It specifically targets reading **Lance v2.x** columnar data files and converting them into Apache Arrow `VectorSchemaRoot` objects. The project is a multi-module Maven project designed for cross-language compatibility with the Python Lance implementation.

The project is currently a **reader only** — there is no writer implementation yet.

## Technology Stack

- **Java 17** (source and target)
- **Maven 3** (multi-module build)
- **Apache Arrow 17.0.0** (`arrow-vector`, `arrow-memory-netty`, `arrow-format`)
- **Protocol Buffers 3.25.1** (`protobuf-java`) — for Lance metadata serialization
- **JUnit 5.10.2** (`junit-jupiter`) — testing framework
- **AssertJ 3.25.3** — fluent assertions
- **SLF4J** — logging

## Project Structure

The project is organized as a standard Maven multi-module project:

```
.
├── pom.xml                          # Root parent POM
├── lance-core/
│   ├── pom.xml
│   └── src/main/proto/              # Protobuf definitions (5 schema files)
├── lance-format/
│   ├── pom.xml
│   └── src/main/java/com/github/jlance/format/   # Main reader implementation
├── lance-tests-compat/
│   ├── pom.xml
│   └── src/test/java/com/github/jlance/compat/   # Compatibility tests
├── compat_tests/
│   ├── milestone_01_gen.py ... milestone_39_gen.py   # Python test data generators
│   └── data/                        # Generated Lance test datasets
└── org/apache/arrow/vector/         # Copied Apache Arrow .class overrides (root level)
```

### Module Details

#### `lance-core`
- Contains **protobuf schema files** under `src/main/proto/`:
  - `file.proto` — V1 file metadata (`FileDescriptor`, `Schema`, `Field`)
  - `file2.proto` — **V2 file format layout** (`ColumnMetadata`, `Page`, `Encoding`)
  - `encodings_v2_0.proto` — V2 array encodings (`Flat`, `Nullable`, `List`, `Binary`, `Dictionary`, `FixedSizeList`, etc.)
  - `table.proto` — Dataset-level structures (`Manifest`, `DataFragment`, `DataFile`, `DeletionFile`)
  - `rowids.proto` — Row ID encoding formats
  - `transaction.proto` — Transaction messages
- Build uses `protobuf-maven-plugin` to generate Java classes.
- No hand-written Java source files.

#### `lance-format`
- The main **file format reader** implementation.
- **Package:** `com.github.jlance.format` and subpackages (`buffer`, `decoder`).
- **Key classes:**
  - `LanceFileReader` — Main API entry point. Opens a `.lance` file, parses footer, reads metadata and batches.
  - `LanceDatasetReader` — Dataset-level reader. Opens a Lance dataset directory, reads manifest, and merges fragments into a single `VectorSchemaRoot`.
  - `LanceFileFooter` — Immutable parsed footer with version, offsets, and column metadata.
  - `LanceFileMetadata` — Holds Arrow `Schema` and row count.
  - `LanceSchemaConverter` — Converts Lance protobuf `Field` objects into Arrow `Field` / `Schema`.
  - `PageDecoder` — Orchestrates reading page buffers and dispatching to `ArrayDecoder` implementations.
  - `ArrayDecoder` interface and implementations: `FlatDecoder`, `NullableDecoder`, `BinaryDecoder`, `DictionaryDecoder`, `FixedSizeListDecoder`, `ConstantDecoder`.
- **No unit tests** in this module (all tests are in `lance-tests-compat`).

## Build and Test Commands

### Compile the entire project
```bash
mvn compile
```

### Run all tests
```bash
mvn test
```

> ⚠️ **CRITICAL: Always use `mvn clean test` before committing.**
>
> Because `lance-format` is a library dependency of `lance-tests-compat`, an incremental build (`mvn test` without `clean`) may reuse a stale `lance-format-*.jar` from the local Maven repository (`~/.m2`) or the previous build. This can mask compilation errors or test failures in `lance-format` itself, giving a false pass. Commit `b17cdd2` was reverted because it passed under an incremental build but failed under `mvn clean test`.

### Clean and install
```bash
mvn clean install
```

### Run a specific test class
```bash
mvn test -pl lance-tests-compat -Dtest=Milestone01CompatTest
```

### Important: JVM Arguments for Testing
The `lance-tests-compat` module configures `maven-surefire-plugin` with:
```
--add-opens=java.base/java.nio=org.apache.arrow.memory.core,ALL-UNNAMED
```
This is **required on Java 17+** because Apache Arrow memory uses `sun.misc.Unsafe` / `java.nio` internals. If you run tests outside of Maven, you must pass this flag manually.

## Code Style Guidelines

The codebase follows these conventions (inferred from existing source):

- **Indentation:** 2 spaces (not tabs).
- **Imports:** No wildcard imports. Every type is imported explicitly, even from the same package. Fully-qualified inline names are used sparingly (e.g., `org.apache.arrow.vector.types.pojo.Field` appears inline in a few files).
- **Braces:** K&R style (opening brace on the same line).
- **Field modifiers:** `private static final` for constants; `private final` for instance fields.
- **Immutability:** Prefer immutable data classes. Use `Collections.unmodifiableList(...)` for exposed lists. Use Java `record` for simple data carriers (e.g., `ColumnMetadataOffset`).
- **Null safety:** Avoid `null` where possible. Nullable encodings are handled via explicit `NullableDecoder` wrappers.
- **Javadoc:** Write Javadoc for public classes and methods. Use `{@link ...}` for cross-references.
- **Logging:** Use SLF4J (`LoggerFactory.getLogger(...)`).
- **Exceptions:** Use `UnsupportedOperationException` for unimplemented features and `IllegalStateException` / `IllegalArgumentException` for contract violations.
- **Switch expressions:** The project uses modern Java `switch` expressions (introduced in Java 14).

## Testing Instructions

### How the compatibility test suite works
1. **Python generators** (`compat_tests/milestone_*_gen.py`) create golden Lance V2 files using the official Python `lance` library via PyArrow.
2. **Java tests** read those files from `compat_tests/data/` and verify the Java reader produces identical Arrow data.
3. Tests dynamically discover `.lance` files in data directories using `Files.list()`.

### Regenerating test data
If you change the Python generator scripts or need to regenerate golden files, run the Python scripts manually. Requirements:
- Python 3
- `pyarrow`
- `lance` (the Python Lance package)

Example:
```bash
cd compat_tests
python milestone_01_gen.py
```

Note: Milestones 05+ use `lance.file.LanceFileWriter` with `max_page_bytes=1024` to force multi-page files.

### Arrow resource cleanup
Tests and production code must properly close Arrow resources:
- `LanceFileReader` — `AutoCloseable`, closes the `FileChannel`
- `BufferAllocator` — call `close()` when done
- `VectorSchemaRoot` — call `close()` when done
- `FieldVector` / `ValueVector` — call `close()` when done

Failure to close these will leak native memory.

## Architecture Notes

### Lance v2 File Layout
The Lance file format is minimal and little-endian:
- **Data Pages** → arbitrary buffers
- **Column Metadatas** → protobuf messages
- **Column Metadata Offset Table (CMO)** → `(position, size)` pairs
- **Global Buffer Offset Table (GBO)** → `(position, size)` pairs
- **Footer (40 bytes, little-endian):**
  - offset to column metadata
  - offset to CMO table
  - offset to GBO table
  - num global buffers
  - num columns
  - major/minor version
  - magic `"LANC"`

See `file2.proto` for the authoritative specification (it contains detailed ASCII diagrams).

### Version Quirk
Lance v2.0 files may have footer version `(0, 3)` or `(2, 0)` due to historical reasons. The reader handles both (`LanceFileFooter.isV2()`).

### Encoding Tree
Most encodings are layered on top of each other, forming a tree with a single root node. Decoders are created recursively by `PageDecoder.createDecoder(ArrayEncoding)`.

### Multi-column Arrow Types
Structs and lists map to **multiple Lance columns**. The reader tracks column indices with an `int[] nextColIndex` passed down recursively during `readBatch()`.

### Schema Conversion
Lance stores schema as a flat list of `Field` protobuf messages with `parent_id` and a string `logical_type` (e.g., `"int32"`, `"timestamp:us:America/New_York"`, `"decimal:128:10:2"`, `"fixed_size_list:float:768"`). `LanceSchemaConverter` rebuilds the Arrow field hierarchy from this flat list.

## Special Files and Directories

- **`org/apache/arrow/vector/`** at the project root contains **copied `.class` files** from Apache Arrow (e.g., `BaseFixedWidthVector.class`, `StructVector.class`, `VectorSchemaRoot.class`). These appear to be manual overrides or patched copies, not generated by Maven. Do not delete or move them without understanding why they exist.
- **`TestApp.class`**, **`TestList.class`**, **`TempInspect.class`** — compiled classes at root with no corresponding `.java` source. Likely scratch/test artifacts.

## Security Considerations

- The reader opens files via `FileChannel` with `StandardOpenOption.READ` only. No write access is requested.
- All file I/O is unbuffered direct `FileChannel` reads. Be cautious with very large files — the current `readBatch()` implementation reads all rows into memory at once.
- Buffer sizes from the file are trusted and cast to `int` for `ByteBuffer.allocate(...)`. Maliciously crafted files with extremely large buffer sizes could cause `OutOfMemoryError`.
- The `org/apache` directory contains copied third-party `.class` files. If you upgrade Arrow, verify whether these overrides are still needed and update them consistently.

## Known Unimplemented Features

The following features are explicitly marked with `UnsupportedOperationException("not yet supported")` in the codebase:

### V2.1 Page Layouts
- **FullZipLayout variable width** (`bits_per_offset`) — Not supported in `FullZipLayoutDecoder`.
- **FullZipLayout multi-level definition** (`bits_def != 1`) — Only single-bit definition currently supported.

### Repetition Levels (Nested Lists in V2.1)
- **MiniBlockLayout** with `repCompression` — ✅ Supported for up to 2 list layers (`list<list<T>>`).
- **ConstantLayout** with `repCompression` — Not yet supported.
- **FullZipLayout** with `bitsRep > 0` — Not yet supported.

Repetition levels are required for nested lists (e.g. `list_(list_(int32()))`) in the V2.1 format.

### V2.0 Encodings
- (None remaining — all V2.0 encodings are supported.)

### V2.1 Complex Types
- **Nested lists with repetition levels** — MiniBlockLayout supports up to 2 list layers. ConstantLayout and FullZipLayout still throw.
- **Interval, Map, Union** — Not yet supported in `LanceSchemaConverter.parseLogicalType()`.

## Development Workflow Tips

- When adding a new **V2.0** encoding decoder, add it to `PageDecoder.createDecoder()` and implement `ArrayDecoder`.
- When adding a new **V2.1** page layout decoder, add it to `PageDecoder.createPageLayoutDecoder()` and implement `PageLayoutDecoder`.
- When adding support for a new Arrow type, update `LanceSchemaConverter.parseLogicalType()`.
- When adding a new milestone test, create both a Python generator in `compat_tests/` and a Java test class in `lance-tests-compat`.
- Always run the full **`mvn clean test`** suite after changes to `lance-format` because that module has no unit tests of its own. Skipping `clean` risks testing against a stale compiled JAR and hiding real failures.
